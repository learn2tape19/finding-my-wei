# Stewardship Model → IEMS Architecture

## Philosophy: "Preserve Relationships"

### How Architecture Enables This

**CHARTER.md files** document relationships:
- What influences this directory's work?
- What investigations support it?
- Which capabilities enable it?
- Which domains contribute to it?

**Subdirectories** preserve relationship context:
- Work stays close to its origins
- Influence pathways are explicit
- Connections are navigable

**Examples:**

**Publishing Domain Charter** explains:
- Sources from FOUNDER_DOMAIN (Drew's authority)
- Feeds to INTELLECTUAL_ESTATE (completed work)
- Uses MARKETING capability
- Governed by CONSTITUTION

Every relationship is **documented and discoverable**.

---

## Philosophy: "Knowledge Continually Evolves"

### How Architecture Enables This

**Lifecycle States** show evolution:
```
IDEA (FOUNDER_DOMAIN) 
  → RESEARCH (PROJECT_ATLAS)
  → DEVELOPMENT (Domain work)
  → VALIDATION (CAPABILITIES/QUALITY_ASSURANCE)
  → PUBLICATION (Domain + INTELLECTUAL_ESTATE)
  → INFLUENCE (through distribution)
  → LEGACY (permanent preservation)
```

Assets move through states deliberately.

Each state has different governance rules.

**Result:** Knowledge is actively managed, not statically stored.

---

## Philosophy: "Roles Endure, Tools Change"

### How Architecture Enables This

**Stewardship hierarchy is tool-independent:**

```
Founder (Drew)
  ↓
Chief Librarian (Currently Claude, but role persists)
  ↓
Chief Strategy Partner (Currently ChatGPT, but role persists)
  ↓
Domain Stewards (Humans or AIs)
```

**If Claude is replaced:**
- Role transfers to successor
- CHARTERS remain unchanged
- Architecture persists
- Knowledge continues

**Result:** The estate survives technology changes.

---

## Philosophy: "Nothing Depends on a Single Technology"

### How Architecture Enables This

**Repository can move:**
- Currently: Local Coding-folder + GitHub
- Could be: Different VCS, different platform
- Must preserve: Folder structure + CHARTER.md pattern

**Relationships are documented, not embedded:**
- References are explicit (documented in CHARTERS)
- Not hidden in code or automation
- Future systems can understand them

**Result:** Knowledge survives platform migration.

# Knowledge Lifecycle → IEMS States

The Stewardship Model defines **9 lifecycle states**:

```
IDEA → CAPTURED → RESEARCH → DEVELOPMENT → VALIDATION → PUBLICATION → PRESERVATION → INFLUENCE → LEGACY
```

The IEMS **labels assets with their state** and provides **appropriate locations**:

| Stewardship State | IEMS Location(s) | Governance |
|---|---|---|
| IDEA | FOUNDER_DOMAIN/PROJECTS | Creator decides |
| CAPTURED | Any domain/PROJECT_ATLAS | Domain/research lead decides |
| RESEARCH | PROJECT_ATLAS | Research methodology applies |
| DEVELOPMENT | Domain work | Domain director decides |
| VALIDATION | Capabilities/QUALITY_ASSURANCE | QA standards apply |
| PUBLICATION | Domain + INTELLECTUAL_ESTATE/BOOKS | Publication workflow applies |
| PRESERVATION | INTELLECTUAL_ESTATE | Librarian preserves |
| INFLUENCE | (Through publication + distribution) | Measured by analytics |
| LEGACY | INTELLECTUAL_ESTATE + Domain history | Permanent record |

**Assets move deliberately through states, not haphazardly.**

# Operating Cycle → IEMS Workflow

The Stewardship Model defines **continuous cycle**:

```
Observe → Capture → Reflect → Investigate → Understand → Create → Publish → Measure → Improve → Observe
```

The IEMS **maps this cycle to architecture**:

| Cycle Stage | IEMS Enabler | Output |
|---|---|---|
| Observe | FOUNDER_DOMAIN, DOMAINS | Raw experience |
| Capture | 01_OPERATING_SYSTEM/MEMORY | Documented observations |
| Reflect | DECISION_LOGS | Reflection captured |
| Investigate | PROJECT_ATLAS | Research initiated |
| Understand | PROJECT_ATLAS | Frameworks developed |
| Create | DOMAINS | Implementation of frameworks |
| Publish | CAPABILITIES/PUBLISHING | Work published |
| Measure | CAPABILITIES/ANALYTICS | Impact measured |
| Improve | DECISION_LOGS + CHARTERS | Systems improve |
| Observe | (Back to top) | Cycle continues |

**The estate continuously learns from itself.**

# Conversation Stewardship → IEMS Policy

**Stewardship Principle:** "Conversations are source material, not assets."

**IEMS Implementation:**

- Conversations stay in working space (not version-controlled)
- Only refined output enters the estate
- Transformation process: Conversation → Decision Log → Framework → Publication
- Source conversation is documented but not stored as-is

**Example flow:**
```
Conversation about teaching methodology
  ↓ (Refined into)
FOUNDER_DOMAIN decision note
  ↓ (Synthesized into)
EDUCATION_DOMAIN methodology
  ↓ (Validated through)
QUALITY_ASSURANCE process
  ↓ (Published as)
INTELLECTUAL_ESTATE/FRAMEWORKS/teaching
  ↓ (Measured for)
Student success metrics
  ↓ (Feeds back into)
Next conversation
```

# What This Means in Practice

## When Drew Creates Something New

1. It enters FOUNDER_DOMAIN (creation state)
2. CHARTER defines what that domain means
3. Asset is tagged with lifecycle state
4. Relationships are documented
5. When complete, it moves to next stage (research, publication, preservation)
6. Every step is guided by philosophy, governed by charters

## When Claude Organizes Something

1. Claude checks CHARTERS for inclusion criteria
2. Claude preserves all relationships (documents them)
3. Claude maintains version history
4. Claude tags lifecycle state
5. Claude documents the organizational decision
6. Constitution remains untouched

## When ChatGPT Researches Something

1. Research goes into PROJECT_ATLAS
2. Working materials stay external
3. Findings are synthesized
4. Synthesized work moves to appropriate domain
5. If significant, it eventually reaches INTELLECTUAL_ESTATE
6. Impact is measured
